Naive hybrid design
-------------------
The hybrid implementation is intentionally simple, as allowed by Milestone 2.

Design:
- Initial bulk-loaded keys are loaded into LIPP.
- New inserted keys are sent to a DynamicPGM buffer.
- For lookups:
  1. Probe DynamicPGM first.
  2. If not found, probe LIPP.
- For range query support:
  - Combine LIPP range query result and DynamicPGM range query result.
  - This is mostly for interface completeness; Milestone 2 uses mixed
    insert/lookup workloads rather than range-query workloads.
- For flushing:
  - Once the DynamicPGM buffer reaches a threshold, iterate over the buffer
    using DynamicPGM iterators.
  - Insert each buffered key into LIPP one-by-one.
  - Reset the DynamicPGM buffer by reconstructing it as empty.

Important notes:
- This is intentionally naive and synchronous.
- There is no asynchronous migration.
- There is no advanced double-buffering.
- There is no attempt to outperform both baselines.
- The design satisfies the "no keys left behind" requirement because keys are
  not dropped; they remain either in the buffer or are inserted into LIPP.


Files changed and purpose
-------------------------
1. competitors/hybrid_pgm_lipp.h
   Purpose:
   - Defines the HybridPGMLIPP competitor class
   - Uses:
     - Lipp<KeyType> as the primary bulk-loaded index
     - DynamicPGMIndex<KeyType,...> as the insert buffer

   Important methods:
   - Build(...)
     - Sorts data if needed
     - Bulk-loads LIPP
     - Resets DynamicPGM buffer
   - EqualityLookup(...)
     - Buffer first, then LIPP
   - Insert(...)
     - Inserts into buffer
     - Flushes when threshold reached
   - FlushBufferToLipp()
     - Iterates over DynamicPGM entries
     - Inserts each entry into LIPP one-by-one
   - size()
     - Returns LIPP size plus DynamicPGM buffer size in bytes

2. benchmarks/benchmark_hybrid_pgm_lipp.h
   Purpose:
   - Declares benchmark_64_hybrid_pgm_lipp overloads

3. benchmarks/benchmark_hybrid_pgm_lipp.cc
   Purpose:
   - Registers preset HybridPGMLIPP variants for Facebook mixed workloads
   - Runs a few candidate combinations of:
     - DPGM error bound
     - flush threshold

   Current preset variants:
   - eps 64, flush 50000
   - eps 128, flush 100000
   - eps 512, flush 200000

4. benchmark.cc
   Purpose:
   - Adds HybridPGMLIPP to the benchmark dispatch
   - Supports:
     --only HybridPGMLIPP

5. CMakeLists.txt
   Purpose:
   - Includes the hybrid benchmark source in benchmark build
   - Also adjusted for lower dependency assumptions in this environment

6. scripts/create_minimal_cmake.sh
   Purpose:
   - Ensures the generated minimal CMake includes hybrid benchmark source too

7. scripts/run_milestone2.sh
   Purpose:
   - Runs exactly the Milestone 2 Facebook mixed workloads
   - Rewrites CSV header after benchmark generation

8. scripts/analysis_milestone2.py
   Purpose:
   - Reads the two Facebook mixed workload result CSVs
   - Computes average throughput across 3 repeats
   - Selects the best row per index_name
   - Writes:
     - analysis_results/milestone2_all_configs.csv
     - analysis_results/milestone2_best_configs.csv
     - analysis_results/milestone2_summary_table.csv
     - analysis_results/milestone2_throughput.png
     - analysis_results/milestone2_index_size.png
     - analysis_results/milestone2_report.md

9. benchmark.h and searches/search.h
   Purpose:
   - Replaced Boost chrono timing with std::chrono to reduce build dependency

10. dtl/adept.hpp
    Purpose:
    - Replaced Boost aligned allocator dependency with a local aligned allocator
    - This was needed because the current cloud image lacks Boost headers

Exact workflow to finish Milestone 2 on cluster

Assume execution from repository root.

--------------------------

- Use Adroit or Della/cluster-style environment
- Load Anaconda
- Run benchmarks in a job rather than an interactive login node

For Neuronic or a similar cluster, adapt the module commands as needed.

  module load anaconda3/2023.3
  module load gcc
  export CC=gcc
  export CXX=g++

Expected hybrid behavior
------------------------
Do not expect the naive hybrid to beat both baselines.

Why:
- The hybrid pays two lookup checks in the worst case:
  buffer, then LIPP
- The flush is synchronous
- The flush inserts buffered keys into LIPP one-by-one
- LIPP does not support incremental bulk-loading into an already-populated index

The hybrid is still valid for Milestone 2 because:
- It combines DPGM and LIPP correctly
- It preserves keys
- It measures the required workloads
- It provides reportable throughput and index-size comparisons
